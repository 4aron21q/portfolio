Контроллер для опроса электросчетчика Меркурий 202 на C

Учебный проект по работе с последовательным портом и бинарными протоколами обмена данными. Реализует базовый опрос счетчика электроэнергии через оптический порт или интерфейс RS-485.

Компиляция:
gcc mercury_ctrl.c -o mercury_ctrl

Запуск:
./mercury_ctrl

Требования:
- ОС Linux или WSL
- Права доступа к последовательному порту (добавьте пользователя в группу dialout или запустите с sudo)
- Подключенный счетчик Меркурий 202 через USB-RS485 адаптер или оптопорт

Настройка:
- Устройство: /dev/ttyUSB0 (изменить в коде при необходимости)
- Скорость: 9600 бод
- Формат данных: 8 бит, без четности, 1 стоп-бит (8N1)
- Режим: raw (без обработки управляющих символов)

Как это работает:
- Программа открывает последовательный порт и настраивает параметры соединения через termios.
- Формирует запрос по протоколу Меркурий: адрес устройства, код команды, контрольная сумма.
- Отправляет запрос в порт и ожидает ответ с таймаутом.
- Принятые данные выводятся в консоль в шестнадцатеричном формате для последующего разбора.
- Цикл опроса повторяется каждые 5 секунд.

Структура протокола (упрощенно):
- Байт 0: Адрес устройства (0x01 для первого счетчика в шине)
- Байт 1: Код команды (0x06 чтение параметров)
- Байт 2: Контрольная сумма (простая сумма байтов по модулю 256)

Особенности реализации:
- Использует только стандартные POSIX-вызовы (open, read, write, tcsetattr)
- Реализован расчет контрольной суммы для целостности данных
- Настройка порта в режим raw для передачи бинарных данных без искажений
- Обработка ошибок открытия порта и чтения данных

Важно:
- Код создан в образовательных целях для изучения работы с последовательными интерфейсами и промышленными протоколами.
- Перед запуском убедитесь, что имеете право на взаимодействие с целевым устройством.
- Для парсинга реальных показаний (кВт·ч, напряжение) необходимо доработать декодер ответа согласно полной спецификации протокола Меркурий.
- Простая контрольная сумма не защищает от преднамеренных помех; в реальных системах следует использовать CRC-16.

Возможные направления доработки:
- Добавление парсинга конкретных параметров (энергия по тарифам, мгновенные значения)
- Поддержка адресации нескольких счетчиков в одной шине
- Реализация полноценного CRC-16 вместо простой суммы
- Логирование данных в файл или отправка по сети
- Вынос конфигурации (порт, скорость, адрес) в аргументы командной строки или конфиг-файл
